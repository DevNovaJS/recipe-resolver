## ADDED Requirements

### Requirement: 레시피 필수/선택 재료 구분
시스템은 레시피-재료 매핑에서 재료별로 필수(`is_essential = true`)와 선택(`is_essential = false`)을 구분해야 한다(SHALL). 추천 커버율 계산은 필수재료만을 대상으로 해야 한다(SHALL). 선택재료는 응답에 "있으면 더 좋은 재료"로 별도 표시해야 한다(SHALL).

#### Scenario: 필수재료 기준 커버율
- **WHEN** 레시피에 필수재료 4개, 선택재료 2개가 있고 사용자가 필수재료 4개를 모두 보유한 경우
- **THEN** 시스템은 해당 레시피를 완벽 매칭으로 분류한다 (선택재료 보유 여부와 무관)

#### Scenario: 선택재료 힌트 표시
- **WHEN** 사용자가 추천 응답을 받은 경우
- **THEN** 각 레시피별로 선택재료 목록과 사용자 보유 여부를 함께 반환한다

### Requirement: 완벽 매칭 추천
시스템은 사용자 보유 재료만으로 필수재료를 모두 충족하는 레시피(필수재료 커버율 100%)를 추천해야 한다(SHALL).

#### Scenario: 완벽 매칭 레시피 존재
- **WHEN** 사용자가 보유 재료 [토마토, 양파, 소고기]로 추천 요청하고 해당 재료로 필수재료가 전부 충족되는 레시피가 있는 경우
- **THEN** 시스템은 해당 레시피 목록을 반환한다

#### Scenario: 완벽 매칭 레시피 없음
- **WHEN** 사용자가 보유 재료로 추천 요청했으나 필수재료 100% 매칭 레시피가 없는 경우
- **THEN** 시스템은 빈 완벽 매칭 목록을 반환한다

### Requirement: 거의 매칭 추천
시스템은 필수재료 1~2개만 추가 구매하면 만들 수 있는 레시피를 추천해야 한다(SHALL). 부족 필수재료를 하이라이트하여 표시해야 한다(SHALL).

#### Scenario: 거의 매칭 레시피 존재
- **WHEN** 사용자가 보유 재료로 추천 요청
- **THEN** 시스템은 부족 필수재료 1~2개인 레시피 목록을 부족 재료 정보와 함께 반환한다

#### Scenario: 부족 재료 수 기준 정렬
- **WHEN** 사용자가 거의 매칭 추천을 요청
- **THEN** 시스템은 부족 필수재료 수가 적은 순으로 정렬하여 반환한다

### Requirement: 레시피 매칭 커버율 계산
시스템은 `recipe_ingredient` 매핑 테이블과 `user_ingredient` 테이블을 `ingredient.id` 정수 FK로 조인하여 커버율을 계산해야 한다(SHALL). 계산은 `is_essential = true`인 재료에 한해야 한다(SHALL).

#### Scenario: 커버율 계산
- **WHEN** 레시피의 필수재료가 5개이고 사용자가 3개를 보유한 경우
- **THEN** 시스템은 해당 레시피의 커버율을 60%로 계산한다 (선택재료는 계산에서 제외)

#### Scenario: 보유 재료 0개
- **WHEN** 사용자의 보유 재료가 0개인 경우
- **THEN** 시스템은 모든 레시피의 커버율을 0%로 계산하고 빈 추천 목록을 반환한다

#### Scenario: 이름 대신 ID 매칭
- **WHEN** 동일한 재료 사전 ID를 참조하지만 재료 이름이 변경된 경우
- **THEN** 시스템은 이름과 무관하게 `ingredient.id` 기반으로 정확히 매칭한다

### Requirement: LLM 창의적 추천
시스템은 LLM API를 연동하여 DB에 없는 창의적 레시피를 추천해야 한다(SHALL).

#### Scenario: LLM 추천 정상 응답
- **WHEN** 사용자가 보유 재료로 LLM 추천을 요청
- **THEN** 시스템은 LLM API에 보유 재료 목록을 전달하고, 응답 받은 창의적 레시피를 반환한다

#### Scenario: LLM API 응답 지연 (타임아웃)
- **WHEN** LLM API 응답이 10초를 초과하는 경우
- **THEN** 시스템은 타임아웃 처리하고 DB 기반 추천 결과만 반환하며, LLM 추천 불가 메시지를 포함한다

#### Scenario: LLM API 서킷 오픈
- **WHEN** LLM API 실패율이 50%를 초과하여 서킷이 오픈된 경우
- **THEN** 시스템은 LLM 호출을 시도하지 않고 즉시 DB 기반 추천 결과만 반환하며, LLM 추천 일시 불가 메시지를 포함한다

### Requirement: LLM 응답 Redis 캐싱
시스템은 동일 재료 조합에 대한 LLM 응답을 Redis에 캐싱해야 한다(SHALL). 캐시 키는 재료 사전 ID를 정렬한 후 해시한 값을 사용해야 한다(SHALL).

#### Scenario: 캐시 히트
- **WHEN** 동일 재료 조합으로 LLM 추천을 요청한 경우 (재료 순서 무관)
- **THEN** 시스템은 LLM API를 호출하지 않고 캐시된 결과를 반환한다

#### Scenario: 캐시 미스
- **WHEN** 캐시에 없는 새로운 재료 조합으로 LLM 추천을 요청한 경우
- **THEN** 시스템은 LLM API를 호출하고, 응답을 캐시에 저장한 후 결과를 반환한다

#### Scenario: 캐시 TTL 만료
- **WHEN** 캐시된 LLM 응답의 TTL(24시간)이 만료된 경우
- **THEN** 시스템은 LLM API를 다시 호출하여 새로운 응답을 캐싱한다

### Requirement: LLM Rate Limiting
시스템은 LLM API 호출에 대해 분당 호출 수를 제한해야 한다(SHALL).

#### Scenario: Rate Limit 이내 호출
- **WHEN** 분당 호출 수가 제한(10회) 이내인 경우
- **THEN** 시스템은 정상적으로 LLM API를 호출한다

#### Scenario: Rate Limit 초과
- **WHEN** 분당 호출 수가 제한(10회)을 초과한 경우
- **THEN** 시스템은 LLM 호출을 거부하고 DB 기반 추천 결과만 반환하며, Rate Limit 초과 메시지를 포함한다
