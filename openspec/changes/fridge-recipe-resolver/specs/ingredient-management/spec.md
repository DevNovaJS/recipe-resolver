## ADDED Requirements

### Requirement: 재료 사전 구성
시스템은 재료 사전(master) 테이블을 별도로 운영해야 하며, 재료 사전은 대표 이름(UNIQUE), 카테고리, 기본 단위를 가진다(SHALL). 사용자가 보유하는 재료와 레시피가 요구하는 재료는 모두 이 재료 사전을 정수 FK로 참조해야 한다(SHALL).

#### Scenario: 초기 시드 데이터 적재
- **WHEN** 애플리케이션이 최초 기동되거나 시드 스크립트가 실행된 경우
- **THEN** 시스템은 한국 가정식 기준 재료 사전을 카테고리별로 사전 적재한다

#### Scenario: 재료 사전 이름 유일성
- **WHEN** 이미 존재하는 대표 이름과 동일한 이름으로 재료 사전 추가가 시도된 경우
- **THEN** 시스템은 중복 등록을 거부하고 400 에러를 반환한다

### Requirement: 재료 자동완성
시스템은 사용자가 재료를 입력할 때 재료 사전을 기반으로 한 자동완성을 제공해야 한다(SHALL). 사용자는 자유 텍스트로 재료를 등록할 수 없으며, 자동완성 결과에서 선택해야 한다(SHALL).

#### Scenario: 접두어 기반 자동완성
- **WHEN** 사용자가 "토마"를 입력하여 자동완성 요청
- **THEN** 시스템은 "토마토", "방울토마토" 등 접두어와 일치하는 재료 사전 항목을 반환한다

#### Scenario: 자동완성 결과 없음
- **WHEN** 사용자가 재료 사전에 없는 접두어를 입력
- **THEN** 시스템은 빈 목록을 반환한다

### Requirement: 사용자 재료 등록
시스템은 사용자가 재료 사전에서 선택한 재료를 자신의 보유 재고로 등록할 수 있어야 한다(SHALL). 등록 시 수량, 단위, 유통기한을 포함해야 한다(SHALL).

#### Scenario: 사용자 재료 정상 등록
- **WHEN** 사용자가 재료 사전 ID(토마토), 수량(5), 단위("개"), 유통기한("2026-04-20")을 입력하여 등록 요청
- **THEN** 시스템은 해당 재료 사전을 참조하는 사용자 보유 재고 row를 생성하고 반환한다

#### Scenario: 존재하지 않는 재료 사전 ID
- **WHEN** 사용자가 재료 사전에 없는 ID로 사용자 재료 등록 요청
- **THEN** 시스템은 404 에러를 반환한다

#### Scenario: 필수 값 누락 시 등록 실패
- **WHEN** 사용자가 수량 없이 사용자 재료 등록 요청
- **THEN** 시스템은 400 에러와 함께 누락 필드 정보를 반환한다

#### Scenario: 동일 재료·동일 단위 중복 등록
- **WHEN** 사용자가 이미 보유 중인 재료 사전 ID와 동일 단위로 재등록을 요청
- **THEN** 시스템은 기존 사용자 재료의 수량을 합산하여 업데이트한다

#### Scenario: 동일 재료·다른 단위 등록
- **WHEN** 사용자가 이미 보유 중인 재료 사전 ID를 다른 단위로 등록 요청
- **THEN** 시스템은 별개 row로 저장한다 (단위 변환을 수행하지 않음)

### Requirement: 카테고리 분류
시스템은 재료 사전을 사전 정의된 카테고리(육류, 채소, 양념, 유제품, 냉동식품 등)로 분류해야 한다(SHALL). 카테고리는 flat ENUM으로 관리되며 계층 구조를 가지지 않는다(SHALL).

#### Scenario: 카테고리별 재료 사전 조회
- **WHEN** 사용자가 카테고리 "채소"로 재료 사전 목록 조회
- **THEN** 시스템은 해당 카테고리에 속한 재료 사전 항목을 반환한다

### Requirement: 사용자 재료 수량 수정
시스템은 등록된 사용자 재료의 수량을 수정할 수 있어야 한다(SHALL).

#### Scenario: 수량 정상 수정
- **WHEN** 사용자가 사용자 재료 ID와 새로운 수량(3)으로 수정 요청
- **THEN** 시스템은 해당 사용자 재료의 수량을 업데이트하고 변경된 정보를 반환한다

#### Scenario: 존재하지 않는 사용자 재료 수정
- **WHEN** 사용자가 존재하지 않는 사용자 재료 ID로 수정 요청
- **THEN** 시스템은 404 에러를 반환한다

### Requirement: 사용자 재료 삭제
시스템은 등록된 사용자 재료를 삭제할 수 있어야 한다(SHALL).

#### Scenario: 사용자 재료 정상 삭제
- **WHEN** 사용자가 사용자 재료 ID로 삭제 요청
- **THEN** 시스템은 해당 사용자 재료를 삭제하고 204 응답을 반환한다

#### Scenario: 존재하지 않는 사용자 재료 삭제
- **WHEN** 사용자가 존재하지 않는 사용자 재료 ID로 삭제 요청
- **THEN** 시스템은 404 에러를 반환한다

### Requirement: 유통기한 임박 재료 조회
시스템은 유통기한이 임박한 사용자 재료 목록을 조회할 수 있어야 한다(SHALL). 기준일은 D-3, D-1, 당일 중 선택 가능해야 한다(SHALL).

#### Scenario: D-3 기준 임박 재료 조회
- **WHEN** 사용자가 기준 D-3으로 유통기한 임박 재료 조회 요청
- **THEN** 시스템은 유통기한이 3일 이내인 사용자 재료 목록을 유통기한 오름차순으로 반환한다

#### Scenario: 임박 재료 없음
- **WHEN** 사용자가 유통기한 임박 재료 조회 요청했으나 해당 재료가 없는 경우
- **THEN** 시스템은 빈 목록을 반환한다

#### Scenario: 유통기한 경과 재료 포함
- **WHEN** 사용자가 유통기한 임박 재료 조회 요청
- **THEN** 시스템은 유통기한이 이미 경과한 재료도 함께 반환하며, 경과 여부를 플래그로 표시한다

### Requirement: 사용자 재료 목록 조회
시스템은 등록된 사용자 재료 전체 목록을 조회할 수 있어야 한다(SHALL). 카테고리별 필터링이 가능해야 한다(SHALL).

#### Scenario: 전체 사용자 재료 조회
- **WHEN** 사용자가 사용자 재료 목록 조회 요청
- **THEN** 시스템은 등록된 모든 사용자 재료를 재료 사전 정보와 함께 반환한다

#### Scenario: 카테고리 필터링 조회
- **WHEN** 사용자가 카테고리 "채소"로 필터링하여 조회 요청
- **THEN** 시스템은 재료 사전의 카테고리가 "채소"인 사용자 재료만 반환한다
